# ROS2の仕組み
[Configuring environment — ROS 2 Documentation: Humble documentation](https://docs.ros.org/en/humble/Tutorials/Beginner-CLI-Tools/Configuring-ROS2-Environment.html)

## ROS2
ROS 2は、シェル環境を用いてワークスペースを"結合"するという概念を採用しています。

## ワークスペース
ROS用語で、ROS 2で開発を行っているシステム上の場所（ディレクトリ）を指します。
ROS 2のコアとなるワークスペースはアンダーレイと呼ばれます。
それに続くローカルワークスペースはオーバーレイと呼ばれます。
ROS 2で開発を行う場合、通常、複数のワークスペースを同時にアクティブにすることになります。
ワークスペースを統合することで、異なるバージョンのROS 2や異なるパッケージセットに対する開発が容易になります。
また、複数のROS 2ディストリビューション（「ディストロ」、例えばDashingやEloquentなど）を同じコンピューターにインストールし、それらを切り替えて使用することも可能です。

## ROS2グラフ
次の数回のチュートリアルでは、「ROS 2グラフ」と呼ばれる一連のROS 2のコア概念について学びます。
ROSグラフは、ROS 2の各要素が同時にデータを処理するネットワークです。
すべての実行可能ファイル（ノード）とそれらの間の接続を網羅し、それらをすべてマッピングして視覚化します。

## 追加の環境設定
### ROS_DOMAIN_ID環境変数の設定（必要な場合）
ROS ドメイン ID の詳細については、ドメインID の記事を参照してください。

### ROS_LOCALHOST_ONLY環境変数の設定（必要な場合）
デフォルトでは、ROS 2の通信はlocalhostに限定されません。 
`ROS_LOCALHOST_ONLY`環境変数を設定することで、ROS 2の通信をlocalhostのみに制限できます。
つまり、ROS 2システムとそのトピック、サービス、アクションは、ローカルネットワーク上の他のコンピュータからは見えなくなります。
教室など、複数のロボットが同じトピックにパブリッシュすることで予期せぬ動作を引き起こす可能性がある環境では、`ROS_LOCALHOST_ONLY`環境変数の設定が役立ちます。
環境変数は次のコマンドで設定できます。
```bash
echo "export ROS_LOCALHOST_ONLY=1" >> ~/.bashrc
```

# ノードの理解
[Understanding nodes — ROS 2 Documentation: Humble documentation](https://docs.ros.org/en/humble/Tutorials/Beginner-CLI-Tools/Understanding-ROS2-Nodes/Understanding-ROS2-Nodes.html)

## ROS2のノード
ROSの各ノードは、例えばホイールモーターの制御やレーザー距離計からのセンサーデータのパブリッシュなど、ロボットシステム内で単一のモジュール化された目的を担当します。
各ノードは、トピック、サービス、アクション、またはパラメータを介して他のノードとデータを送受信できます。

![ROS2 Nodes](./image/ros2_nodes.gif)

完全なロボットシステムは、連携して動作する多数のノードで構成されています。
ROS 2では、単一の実行可能ファイル（C++プログラム、Pythonプログラムなど）に1つ以上のノードを含めることができます。

## ROS2の実行
このコマンドはパッケージから実行可能ファイルを起動します。

```bash
ros2 run <package_name> <executable_name>
```
しかし、実行しただけでは、ノード名はまだ分かりません。
ノード名を見つけるには、以下のコマンドを実行します。

## ノードリスト
実行中のすべてのノードの名前を表示します。
これは、ノードとやり取りしたい場合や、システムで多数のノードが動作していてそれらを管理する必要がある場合に特に便利です。
```bash
ros2 node list
```

## リマッピング
再マッピングを使用すると、ノード名、トピック名、サービス名などのデフォルトのノードプロパティをカスタム値に再割り当てできます。
それでは、ノードの名前を変更してみましょう。
```bash
ros2 run <package_name> <executable_name> --ros-args --remap __node:=<new_node_name>
```

## ノード情報
ノードの名前がわかったので、次のコマンドでノードの詳細情報にアクセスできます。

```bash
ros2 node info <node_name>
```
サブスクライバー、パブリッシャー、サービス、およびアクションのリストを返します。
つまり、そのノードと対話する ROS グラフ接続のリストを返します。

# トピックの理解

## ROS2のトピック
ROS 2は複雑なシステムを多数のモジュールノードに分割します。
トピックはROSグラフの重要な要素であり、ノード間のメッセージ交換のためのバスとして機能します。
ノードは任意の数のトピックにデータをパブリッシュでき、同時に任意の数のトピックをサブスクリプションできます。

![ROS2 Topics](./image/ros2_topics.gif)

トピックは、データがノード間、つまりシステムのさまざまな部分間で移動される主な方法の 1 つです。

## トピックリスト
システムで現在アクティブなすべてのトピックのリストが返されます。
```bash
ros2 topic list
```

トピックの種類を括弧内に追加表示したい場合、`--show-types`(または`-t`) フラグを設定します。
```bash
ros2 topic list -t
```

これらの属性、特にタイプは、トピック間を移動するときにノードが同じ情報について話していることを認識する方法です。

## トピックエコー
トピックにパブリッシュされているデータを表示します。

```bash
ros2 topic echo <topic_name>
```

## トピック情報
トピックは 1 対 1 の通信である必要はなく、1 対多、多対 1、多対多の通信も可能です。
これを別の方法で見るには、次のコマンドでトピックの詳細情報にアクセスします。

```bash
ros2 topic info <topic_name>
```
トピックのさらなる詳細情報を取得するには、`--verbose`(または`-v`) フラグを使用します。

```bash
ros2 topic info <topic_name> -v
```
これにより、次のような追加の詳細が返されます:

- パブリッシャーとサブスクライバーのノード名と名前空間
- トピックの種類
- QoSプロファイル

## トピックパブ
メッセージ構造ができたので、次のコマンドを使用して、コマンド ラインから直接トピックにデータをパブリッシュできます。
この引数はYAML構文で入力する必要があることに注意してください。
継続的に動作させるには、一定のコマンドストリームが必要です。
コマンドライン オプションがない場合、コマンドは 1 Hz で一定の ストリームで発行されます。
```bash
ros2 topic pub <topic_name> <msg_type> '<{arg1: {value1: , value2: ,}, arg2: {value1: , value2: ,}, }>'
```

トピックにデータを一度だけ（継続的にではなく）パブリッシュしたい場合があります。コマンドを一度だけパブリッシュするには、`--once`オプションを追加してください。
```bash
ros2 topic pub --once -w 2 <topic_name> <msg_type> '<{arg1: {value1: , value2: ,}, arg2: {value1: , value2: ,}, }>'
```

`--once`は、「メッセージを 1 つパブリッシュしてから終了する」ことを意味するオプションの引数です。
`-w 2`は「2つの一致するサブスクリプションを待つ」という意味のオプション引数です。トピックechoと両方をサブスクリプションしているため、これが必要になります。

## トピック検索
次のコマンドを使用すると、メッセージタイプを指定したときに利用可能なトピックが出力されます。
```bash
ros2 topic find <msg_type>
```

# サービスの理解

## ROS2のサービス
サービスは、ROSグラフ内のノード間の通信におけるもう一つの手段です。
サービスは、トピックのパブリッシャー・サブスクライバーモデルとは対照的に、リクエストとレスポンスのモデルに基づいています。
トピックはノードがデータストリームをサブスクライブして継続的な更新情報を取得できるようにしますが、サービスはクライアントから明示的に呼び出された場合にのみデータを提供します。
このチュートリアルをご覧ください。Robotis ロボット アームを使用した ROS サービスの優れた現実的なアプリケーションです。

![ROS2 Services](./image/ros2_services.gif)

## サービスリスト
システムで現在アクティブなすべてのサービスのリストが返されます。
```bash
ros2 service list -t
```

ROS 2のほぼすべてのノードには、パラメータ（の基盤となるインフラストラクチャ）サービスがあります。

## サービスタイプ
サービスには、リクエストとレスポンスのデータ構造を表す型があります。サービス型はトピック型と同様に定義されますが、リクエスト用のメッセージとレスポンス用のメッセージの2つの部分で構成されます。


### 型の例
Emptyタイプは、サービス呼び出しが要求時にデータを送信せず、応答を受信するときにデータを受信しないことを意味します。

## サービスコール
サービス タイプとは何か、サービスのタイプを見つける方法、そのタイプの引数の構造を見つける方法がわかったので、次のようにしてサービスを呼び出すことができます。
この引数はYAML構文で入力する必要があることに注意してください。
```bash
ros2 service call <service_name> <service_type> <"{arg1: value, arg2: 'string', }">
```

## インターフェース表示
ノードはメッセージを使用してトピックを介してデータを送信します。
パブリッシャーとサブスクライバーは通信するために同じ種類のメッセージタイプを送受信する必要があります。
`ros2 topic list -t`実行後に確認したトピックタイプから、各トピックで使用されているメッセージタイプがわかります。
`ros2 service list -t`実行後に確認したサービスタイプから、各サービスで使用されているメッセージタイプがわかります。
次のコマンドで、型の詳細、具体的にはメッセージがどのようなデータ構造を期待しているかを知ることができます。

```bash
ros2 interface show <msg_name>
```
トピックの場合、`ros2 topic echo <topic_name>`コマンドで渡されたデータを思い出してください。同じ構造になっています。
サービスの場合、`---`はリクエスト構造（上）とレスポンス構造（下）を分離します。
